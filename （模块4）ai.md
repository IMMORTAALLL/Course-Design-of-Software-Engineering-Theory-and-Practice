# AI 使用记录

## 模块四：AI辅助测试与调试

### 一、测试范围梳理

#### 原始提示词

```text
按照学校课程设计指导书的模块四要求，对当前“智投社区”项目做 AI 辅助测试与调试。先不要写最终报告，先检查仓库里已有的前后端代码，确认可以测试哪些内容，重点关注后端 FastAPI 接口、前端 Vue 页面，以及成员 D 负责的后台审核管理模块。
```

#### AI输出摘要

AI 先读取仓库结构和现有文档，确认当前项目采用 FastAPI + SQLAlchemy 后端、Vue 3 + Vite + TypeScript 前端。随后根据代码目录和路由文件整理出本次测试重点：

- 后端基础检查：Python 编译、FastAPI 应用启动、健康检查接口
- 论坛公开接口：板块列表、帖子列表、热门帖子
- 后台管理接口：概览、审核队列、举报列表、敏感词、用户处置、统计数据
- 后台操作接口：审核结果更新、举报结果更新、敏感词启停
- 前端页面：论坛首页、板块页、热榜页、后台首页、审核页、举报页、敏感词页、用户处置页和统计页

#### 人发现可能的问题

最开始不能只凭“构建通过”判断模块四完成，因为课程设计测试部分需要体现实际测试与调试过程。如果只写命令结果，缺少测试范围、测试对象和问题处理记录。

#### 迭代优化

把测试范围从单纯构建检查扩展为“后端编译 + 后端真实 HTTP 冒烟 + 前端构建 + 浏览器页面冒烟 + 后台按钮操作”。这样既能覆盖运行链路，也能对应成员 D 的后台审核管理模块。

### 二、后端接口测试

#### 原始提示词

```text
对后端做实际测试。不要只看代码，要运行验证。先检查语法，再启动 FastAPI 服务，测试 /api/health、论坛公开接口和 /api/admin 下的后台管理接口。测试时注意不要污染仓库里的数据库文件。
```

#### AI输出摘要

AI 执行了后端编译检查：

```bash
python -m compileall -q backend\app
```

编译检查通过。随后使用临时 SQLite 数据库启动 Uvicorn 服务，并通过 HTTP 请求测试以下接口：

- `/api/health`
- `/api/sections`
- `/api/posts?page=1&size=5`
- `/api/posts/hot?limit=5`
- `/api/admin/overview`
- `/api/admin/audit-queue`
- `/api/admin/reports`
- `/api/admin/sensitive-words`
- `/api/admin/user-moderation`
- `/api/admin/statistics`
- `/api/admin/audit-queue/{id}`
- `/api/admin/reports/{id}`
- `/api/admin/sensitive-words/{id}`

测试结果显示，上述接口均返回 HTTP 200，统一响应中的 `code` 为 0，并且响应体中包含 `data` 字段。

#### 人发现可能的问题

测试过程中发现系统默认 Python 为 3.13.2，而项目依赖目录中的 `pydantic_core` 是 Python 3.12 对应的二进制文件。直接使用系统 Python 运行 FastAPI 相关测试时，会出现 `ModuleNotFoundError: No module named 'pydantic_core._pydantic_core'`。

#### 迭代优化

改用 Python 3.12 运行后端测试，并设置：

```text
PYTHONPATH=.;venv_lib
DATABASE_URL=sqlite:///临时测试数据库
```

另外，尝试使用 FastAPI `TestClient` 时发现缺少 `httpx` 依赖。为了不临时修改项目依赖，测试方式调整为启动真实 Uvicorn 服务，再通过 HTTP 请求完成接口测试。该方式更接近实际运行环境。

#### 最终结果

后端编译检查和真实 HTTP 接口冒烟测试均通过。测试过程中记录了 Python 版本匹配和 `TestClient` 依赖缺失两个环境问题，为后续安装部署文档提供了补充依据。

### 三、前端构建与页面测试

#### 原始提示词

```text
继续测试前端。先运行前端构建，再用浏览器自动化打开主要页面。重点验证后台管理页面是否能显示数据，审核、举报、敏感词等按钮是否能正常调用接口。
```

#### AI输出摘要

AI 执行前端构建命令：

```bash
npm.cmd run build
```

构建过程包含 `vue-tsc --noEmit` 和 `vite build`，执行结果通过。随后启动 Vite 开发服务，并用 Playwright 访问以下页面：

- `/`
- `/sections`
- `/hot`
- `/admin`
- `/admin/audits`
- `/admin/reports`
- `/admin/sensitive-words`
- `/admin/users`
- `/admin/statistics`

浏览器测试还实际点击了后台页面中的操作按钮：

- 内容审核页点击“通过”
- 举报处理页点击“驳回举报”
- 敏感词管理页点击“停用规则”

#### 人发现可能的问题

前端测试不能只停留在页面打开，还需要确认页面是否真的拿到了后端数据，以及点击操作后是否产生接口请求。否则只能说明路由能访问，不能说明联调通过。

#### 迭代优化

在浏览器自动化测试中增加对页面元素数量的检查，例如后台首页 4 个指标卡片、审核列表 3 行数据、举报列表 2 行数据、敏感词规则 3 个卡片。对敏感词按钮操作前后文案进行了比较，从“停用规则”变为“启用规则”，确认前后端交互生效。

#### 最终结果

前端构建通过，浏览器冒烟测试通过。测试过程中没有出现控制台错误，也没有出现失败的网络请求。后台审核管理模块的页面加载和关键交互均能正常工作。

### 四、测试报告整理

#### 原始提示词

```text
根据刚才实际运行的测试结果，整理模块四测试报告。格式和语言要尽量参考前面模块文档，不要写成完全脱离项目的模板，要体现本项目真实测过什么、发现了什么问题、最后怎么处理。
```

#### AI输出摘要

AI 根据实际测试记录整理出模块四测试报告，内容包括：

- 测试目标
- 测试环境
- 测试范围
- 后端基础检查
- 公开内容接口检查
- 后台管理接口检查
- 前端构建测试
- 浏览器冒烟测试
- 前端交互测试
- 发现的问题与处理
- 测试结论

#### 人发现可能的问题

如果测试报告只写“全部通过”，会显得不真实，也不能体现 AI 辅助调试的过程。因此需要把 Python 版本不匹配、`TestClient` 缺少 `httpx` 这些真实问题写进去。

#### 迭代优化

在测试报告中补充环境问题和处理方式，明确说明测试时使用临时 SQLite 数据库，避免污染仓库内本地数据库文件。同时保留后续可扩展测试方向，例如补充正式接口测试用例、前端回归测试脚本和异常场景测试。

#### 最终结果

模块四测试文档能够反映真实测试过程。文档既记录了通过的测试项，也记录了调试过程中发现的环境问题，能够作为课程设计“AI辅助测试与调试”部分的提交材料。
